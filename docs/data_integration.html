Introduction to Data Integration in R

Data analysis often requires combining data from multiple sources, such as files, databases, APIs, or web scraping. R is a powerful and flexible tool for data integration, but it can also pose some challenges and pitfalls. In this workshop, you will learn some of the best ways to integrate data from multiple sources in R.

Getting a Climate Data Store (CDS) API key

Later in this workshop you’ll download real historical climate data (ERA5) from the Copernicus Climate Data Store (CDS). This requires a free personal account and API key, separate from anything else in this course, so set it up now — it can take a few minutes for the account to activate.

  1. Create a free account at https://cds.climate.copernicus.eu/.
  2. Once logged in, open your user profile page and copy your Personal Access Token — this is your API key.
  3. In R, register it once per computer (you’ll install the ecmwfr package later in this workshop, but you can do it now too):
install.packages("ecmwfr")
library(ecmwfr)
wf_set_key(key = "paste-your-personal-access-token-here")

ecmwfr stores the key in your system’s secure credential manager, so you only need to do this once — every future R session on this computer will use it automatically.

WarningNever save your API key in a script

Type your key directly into the R console (or a chunk you run once and then delete), not into a .qmd file you save or push to GitHub. Treat it like a password — if it’s ever exposed, regenerate it from your CDS profile page.

  4. Combine tabular and raster data.
  5. Combine tabular data by an ID and time.

Choose the right package

R has many packages that can help you import, merge, and manipulate data from different sources. Some of the most popular and useful ones for tables include readr, dplyr, tidyr, and purrr. These packages are part of the tidyverse, a collection of packages that share a consistent and coherent syntax and philosophy for data analysis. For spatial data, the sf and terra packages are useful.

Type of Data Library
Tabular tidyverse
Vector sf
Raster terra

Use pipes and functions

One of the best features of the tidyverse is the pipe operator (%>%), which allows you to chain multiple functions together and pass the output of one function as the input of the next one. This way, you can create a data integration pipeline that is clear and logical, and that avoids intermediate variables and nested functions.

Check and validate data

Data integration can introduce errors or inconsistencies in your data, such as duplicates, mismatches, outliers, or invalid values. Therefore, it is important to check and validate your data before and after you integrate it from multiple sources. You can use various tools and techniques to do this, such as summary statistics, data visualization, data profiling, or data quality rules.

Use st_intersection to extract the country of each tower site. A straight st_intersection() can return more or fewer rows than you started with (a point can sit exactly on a shared border and match two polygons, or fall outside all of them), so join back on SITE_ID instead of assigning the result directly:

country_lookup <- st_intersection(world_sf, fluxnet_ch4_sf) %>%
  st_drop_geometry() %>%
  select(SITE_ID, name) %>%
  distinct(SITE_ID, .keep_all = TRUE)
Warning: attribute variables are assumed to be spatially constant throughout
all geometries
fluxnet_ch4_sf <- fluxnet_ch4_sf %>%
  left_join(country_lookup, by = "SITE_ID") %>%
  rename(country = name)

Joining tables

We can combine columns from two (or more) tables together. This can be achieved using the join family of functions in dplyr. There are different types of joins that will result in different outcomes.

inner_join() includes all rows that appear in both the first data frame (x) and the second data frame (y).

left_join() returns all rows from x based on matching rows on shared columns in y. right_join() is the companion to left_join(), but returns all rows included in y based on matching rows on shared columns in x.

Validating tower measurements with ERA5 reanalysis climate data

The static climate layer you extracted earlier (GlobalClimate.tif) gives each site a single long-term average — useful for describing a site, but not for matching a specific month. fluxnet_flux already includes tower-measured monthly precipitation (P_F) and temperature (TA_F), recorded by instruments at each site. Here we’ll download ERA5, a global atmospheric reanalysis produced by the European Centre for Medium-Range Weather Forecasts (ECMWF), and compare its monthly values against what the towers actually measured. ERA5 is genuinely global and covers specific months back to 1940, which is why we’re using it here instead of the climate normals from earlier workshops — and the same download will double as the input for your spatial projection workshop.

Register your CDS key for this R session, if you haven’t already (see the setup instructions at the top of this workshop):

Download monthly ERA5 data

We need monthly precipitation and temperature for every year covered by fluxnet_flux:

range(fluxnet_flux$YearMon)
[1] "Jan 2006" "Dec 2018"

Build one request per year (global extent, both variables), then submit them together with wf_request_batch(), which runs several requests at once instead of waiting for each one in turn:

era5_years <- 2006:2018

era5_requests <- lapply(era5_years, function(yr) {
  list(
    dataset_short_name = "reanalysis-era5-single-levels-monthly-means",
    product_type = "monthly_averaged_reanalysis",
    variable = c("2m_temperature", "total_precipitation"),
    year = as.character(yr),
    month = sprintf("%02d", 1:12),
    time = "00:00",
    data_format = "netcdf",
    target = paste0("era5_", yr, ".nc")
  )
})
era5_dir <- file.path("data", "products", "era5")
dir.create(era5_dir, showWarnings = FALSE, recursive = TRUE)

wf_request_batch(request_list = era5_requests, workers = 2, retry = 30, user = "ecmwfr", path = era5_dir)
NoteThis step takes a few minutes

Each request is processed by ECMWF’s queue rather than downloaded instantly — the whole batch (13 years, global, 2 variables) typically finishes in a few minutes, but can take longer if the CDS service is busy. You only need to run it once; the files are saved to data/products/era5/. Don’t raise workers or lower retry to try to speed this up — polling too aggressively gets your requests throttled by CDS (“check your retry rate”) rather than completed faster.

Each year downloads as a zip containing one NetCDF file per variable. Unzip them into per-year folders so files with the same name across years don’t collide:

era5_dir <- file.path("data", "products", "era5")
era5_zips <- list.files(era5_dir, pattern = "[.]zip$", full.names = TRUE)

for (z in era5_zips) {
  unzip(z, exdir = tools::file_path_sans_ext(z))
}

Extract monthly values at each tower site

Name each layer with its year-month so we can pivot the extracted values into a long, joinable format:

names(era5_ppt_monthly) <- as.character(zoo::as.yearmon(terra::time(era5_ppt_raw)))
names(era5_tmean_c) <- as.character(zoo::as.yearmon(terra::time(era5_tmean_raw)))

fluxnet_ch4_sf <- st_transform(fluxnet_ch4_sf, crs = crs(era5_ppt_monthly))

era5_ppt_long <- terra::extract(era5_ppt_monthly, fluxnet_ch4_sf, ID = FALSE) %>%
  mutate(SITE_ID = fluxnet_ch4_sf$SITE_ID) %>%
  pivot_longer(-SITE_ID, names_to = "YearMon", values_to = "era5_p_f") %>%
  mutate(YearMon = zoo::as.yearmon(YearMon))

era5_tmean_long <- terra::extract(era5_tmean_c, fluxnet_ch4_sf, ID = FALSE) %>%
  mutate(SITE_ID = fluxnet_ch4_sf$SITE_ID) %>%
  pivot_longer(-SITE_ID, names_to = "YearMon", values_to = "era5_ta_f") %>%
  mutate(YearMon = zoo::as.yearmon(YearMon))

era5_monthly <- era5_ppt_long %>% left_join(era5_tmean_long, by = c("SITE_ID", "YearMon"))

Join ERA5 onto the flux data, the same way you joined NDVI/EVI earlier:

fluxes_month_era5 <- fluxes_month %>% left_join(era5_monthly, by = c("SITE_ID", "YearMon"))

A tower sits at one exact point and measures its immediate surroundings; ERA5 is a reanalysis model output at ~0.25 degree (~28 km) resolution, so each value represents the average condition across a fairly large grid cell. Expect closer agreement for temperature (which varies smoothly over that distance) than for precipitation (which is often much more localized) — this is exactly the kind of gap to check for before trusting a reanalysis product as a stand-in for missing tower data.
