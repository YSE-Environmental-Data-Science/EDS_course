Data Integration: Building a Methane Flux Modeling Table

From data sources to a model-ready table

The final course project asks you to build a methane flux model. Before you can fit a model, you need to compile a dataset where each row is a clean observation and each column is a response variable, predictor, or useful descriptor.

In this workshop, we will ask:

How do we combine flux measurements, site metadata, vegetation indices, soil, climate, elevation, and reanalysis data into one table that can be used to model methane flux?

The target data product is a site-month modeling table:

  • one row = one tower site in one month;
  • response = methane flux (FCH4_F_gC);
  • dynamic predictors = monthly tower climate, MODIS vegetation indices, ERA5 climate;
  • static predictors = ecosystem type, country, soil, climate normals, elevation.

By the end of this workshop, you will be able to:

  1. Identify whether a data source is tabular, vector, or raster.
  2. Identify whether a predictor is static or dynamic.
  3. Choose join keys for site-only and site-time datasets.
  4. Extract raster values to tower locations.
  5. Join static site attributes to monthly observations.
  6. Join dynamic monthly predictors by SITE_ID and YearMon.
  7. Check joins for missing values, duplicate keys, and row-count changes.
  8. Save a model-ready methane flux dataset.

Data inventory

Before writing code, make a data inventory. Every predictor you use in a model should have a source, units, spatial/temporal resolution, and join key.

Static versus dynamic predictors

Static predictors describe a site and can be repeated across months. Dynamic predictors change through time and must be joined by both site and time. Accidentally joining monthly data only by SITE_ID is one of the easiest ways to create duplicated or incorrect model data.

Getting a Climate Data Store (CDS) API key

Later in this workshop, we will use ERA5 monthly climate data from the Copernicus Climate Data Store (CDS). This requires a free personal account and API key.

  1. Create a free account at https://cds.climate.copernicus.eu/.
  2. Once logged in, open your user profile page and copy your Personal Access Token.
  3. Register it once per computer:
install.packages("ecmwfr")
library(ecmwfr)
wf_set_key(key = "paste-your-personal-access-token-here")
Never save your API key in a script

Type your key directly into the R console, not into a .qmd file you save or push to GitHub. Treat it like a password.

Helper checks for joins

We will use a few small checks repeatedly. These are habits you should use in your final project.

count_missing <- function(data) {
  tibble(variable = names(data),
         missing_count = vapply(data, function(x) sum(is.na(x)), integer(1))) %>%
    arrange(desc(missing_count))
}

Check for duplicate keys:

find_duplicate_keys <- function(data, ...) {
  data %>%
    count(..., name = "n") %>%
    filter(n > 1)
}

ERA5 monthly reanalysis data

The static climate layer (GlobalClimate.tif) gives each site one long-term average. ERA5 gives monthly climate values that can be joined to site-month flux rows.

The download workflow is shown below, but the course data folder already includes processed ERA5 rasters:

  • data/products/era5_ppt_monthly.tif
  • data/products/era5_tmean_monthly.tif

Download ERA5 data

library(ecmwfr)

era5_years <- 2006:2018

era5_requests <- lapply(era5_years, function(yr) {
  list(
    dataset_short_name = "reanalysis-era5-single-levels-monthly-means",
    product_type = "monthly_averaged_reanalysis",
    variable = c("2m_temperature", "total_precipitation"),
    year = as.character(yr),
    month = sprintf("%02d", 1:12),
    time = "00:00",
    data_format = "netcdf",
    target = paste0("era5_", yr, ".nc")
  )
})

era5_dir <- file.path("data", "products", "era5")
dir.create(era5_dir, showWarnings = FALSE, recursive = TRUE)

wf_request_batch(request_list = era5_requests,
                 workers = 2,
                 retry = 30,
                 user = "ecmwfr",
                 path = era5_dir)
Why we use the processed rasters below

ERA5 downloads depend on an external account, API key, and queue. For this workshop, we use processed rasters that have already been downloaded and converted to the right units so the data integration workflow is reproducible in class.

Compare tower-measured precipitation (P_F) with ERA5 precipitation:

ggplot(flux_model_era5, aes(x = P_F, y = era5_p_f)) +

       x = "Tower-measured P_F (mm)",

ggplot(flux_model_era5, aes(x = TA_F, y = era5_ta_f)) +
  geom_point(alpha = 0.45) +
  geom_abline(slope = 1, intercept = 0, col = "red") +
  labs(title = "Tower vs. ERA5 monthly mean temperature",

[1] 0.9889273

A tower measures its immediate surroundings. ERA5 represents average conditions across a much larger grid cell. Expect closer agreement for temperature than precipitation. This is the kind of check you should do before trusting a reanalysis product as a predictor or substitute for missing observations.

dissimilarity_vars <- c("mean_p_f", "mean_ta_f", "mean_vpd_f",

Interpreting dissimilarity

This is a screening analysis. If environmentally similar towers have similar methane flux means or variability, that supports using these predictors in a model. If they do not, it suggests important drivers may still be missing, such as water table depth, wetland type, soil carbon, disturbance history, or measurement design.
